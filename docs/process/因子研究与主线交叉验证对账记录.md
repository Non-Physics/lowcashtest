# 因子研究与主线交叉验证对账记录

本文档用于沉淀这次“因子研究支线 vs 当前主线”的交叉验证结果。

目标不是直接裁决研究支线对错，而是把已经确认的事实、当前疑点、以及需要研究支线再次确认的问题写清楚，避免后续继续口头传递造成偏差。

## 1. 这次对账的核心结论

当前因子研究支线里的 `P2` 对照脚本，并没有与当前项目已经收口的“可信主线”完全同口径。

更准确地说：

- `P2` 对照的是旧主线壳子 [股票策略回测.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测.py)
- 当前项目正式收口的可信主线是 [股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py)

因此：

- `P2` 可以作为“因子候选替换旧主线候选源后是否有提升”的证据
- 但不能直接作为“因子候选已经在当前可信基线主线上验证通过”的证据

## 2. 已确认事实

### 2.1 `P2` 当前加载的是哪个回测壳子

[主线收益对照_P2.py](/mnt/d/量化/同花顺/pythonProject/主线收益对照_P2.py) 中：

- 直接从 [factor_research/data_access.py](/mnt/d/量化/同花顺/pythonProject/factor_research/data_access.py) 导入 `get_base_module`

[factor_research/data_access.py](/mnt/d/量化/同花顺/pythonProject/factor_research/data_access.py) 中：

- `_load_base_strategy()` 明确写死：
  - `strategy_path = PROJECT_ROOT / "股票策略回测.py"`

这意味着：

- `P2` 的 `mainline_default`
- `P2` 的 `mainline_factor_candidate`

两者都是跑在 [股票策略回测.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测.py) 上，而不是 [股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py)。

### 2.2 研究支线文档本身也承认这一点

- 当前读取到的主线参数来自 `股票策略回测.py`

这说明研究支线并不是无意间读错文件，而是确实在用旧主线壳子做对照。

## 3. 旧主线壳子与可信基线版的关键差异

这一步很关键，因为问题不只是“文件名不同”，而是执行语义不同。

### 3.1 盘中退出设置不同

[股票策略回测.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测.py)：

- `ENABLE_INTRADAY_EXIT = True`

[股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py)：

- `ENABLE_INTRADAY_EXIT = False`

这会直接影响：

- 交易次数
- 持仓周期
- 卖出原因
- 回撤和收益曲线

### 3.2 开仓市场约束不同

[股票策略回测.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测.py)：

- 即使 `risk_on` 不满足，也仍然允许开仓，只是在理由里标注“弱市轻仓”

[股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py)：

- `STRICT_RISK_ON_FOR_ENTRY = True`
- 非 `risk_on` 默认不允许开仓

这会显著改变：

- 实际参与市场的日期
- 候选股票进入执行层的频率
- 交易密度

### 3.3 `P2` 替换的不是“排序”，而是整个候选入口

[股票策略回测.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测.py) 原始 `select_candidates()` 会做很多硬过滤：

- `amount_ma20`
- `ret20`
- `close_ma20_gap`
- `atr_ratio`
- `ma20_slope_5`
- ST / 上市状态 / 冷却期等

而 [主线收益对照_P2.py](/mnt/d/量化/同花顺/pythonProject/主线收益对照_P2.py) 中 `_patch_select_candidates()` 的逻辑是：

- 直接使用因子支线提前准备好的候选列表
- 只保留冷却期过滤
- 最后截断到 `TOP_N`

因此 `P2` 不是“旧主线条件不变，只换排序分数”，而更接近：

- 用研究支线构造出的候选宇宙 + 因子分数
- 替换旧主线原本的候选生成器

这会导致：

- 交易次数增加或减少
- 候选源本身口径改变
- 结果不能简单解释为“仅排序优化”

## 4. 当前最稳的解释

目前最合理的解释是：

1. 研究支线有意复用旧主线壳子做 `P2` 对照实验
2. 这个实验的目的更像是：
   - 在旧主线执行框架中，验证因子候选源是否有增益
3. 但研究支线没有跟随后来项目对“主线 = 基线版”的正式收口
4. 所以从现在回看，`P2` 属于：
   - 有意的实验设计
   - 但口径没有收口到当前项目真实主线

换句话说：

- 这不是一个纯偶然 bug
- 但它也不是已经对齐后的最终主线验证

## 5. 我对研究支线的质问

下面这些问题是建议你回到研究支线时逐条确认的。

### 问题 1

`P2` 为什么明确绑定 [股票策略回测.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测.py)，而不是 [股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py)？

请研究支线明确回答：

- 这是当时有意为之的实验设计
- 还是后来主线收口后没有同步更新

### 问题 2

`P2` 设计时，想验证的到底是哪一个命题？

请研究支线从下面几种里明确选一个：

1. 因子候选替换旧主线候选源，是否有增益
2. 因子排序替换旧主线排序，是否有增益
3. 因子候选已经在当前可信基线主线上验证通过

从当前代码看，更像第 `1` 条，不像第 `2` 条，更不能直接声称第 `3` 条。

### 问题 3

`P2` 替换的是“候选排序”，还是“整个候选生成器”？

当前代码显示：

- 替换的是整个 `select_candidates`

请研究支线明确确认：

- 这是不是它原本的实验定义
- 如果是，它是否意识到这会把“候选池变化”和“排序变化”混在一起

### 问题 4

研究支线是否承认：

- 目前 `P2` 结果不能直接外推为“因子方案已经在当前基线主线上验证通过”

如果不承认，请它给出代码证据，证明 `P2` 已经严格对齐到 [股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py)。

### 问题 5

研究支线下一步准备如何补这一层验证？

建议它明确回答：

- 是否计划新增一版真正基于 [股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py) 的对照脚本
- 该脚本是否会做到：
  - 壳子完全使用基线版
  - 只替换候选逻辑
  - 其他时序、市场过滤、退出、成本全部保持不变

## 6. 建议研究支线补的真正验证

建议新增一版新的对照，例如：

- `主线收益对照_P3_基线版.py`

它应该满足：

1. 壳子使用 [股票策略回测_基线版.py](/mnt/d/量化/同花顺/pythonProject/股票策略回测_基线版.py)
2. 数据目录与当前主线完全一致
3. 默认参数与当前基线版完全一致
4. 只替换候选逻辑
5. 不修改：
   - 风险过滤
   - 退出逻辑
   - 成本模型
   - 执行时序

只有这一步做完后，才能更有把握回答：

- 因子候选是否真的适合进入当前可信主线
