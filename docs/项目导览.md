# 项目导览

## 项目目标

本项目的目标不是追求“立刻自动实盘”，而是建立一条可信的完整链路：

1. 数据下载与清洗可信
2. 回测逻辑时序严格正确
3. 基线策略可复现
4. 模拟盘执行可验证
5. 交易自动化边界清楚

6. 再次对账并留痕

  交易状态、信号服务、执行与对账

  - 关键策略脚本

原因是这轮刚把同花顺 PDF 自动保存链收口，优先保证入口和路径稳定，再做重命名/移动更稳。
因此本轮只先建立 `diagnostics/ths/` 作为规范归档位置，根目录入口暂时保留。

## 当前边界

当前系统已经能做到：

- 用基线策略生成次日订单意图
- 用同花顺模拟盘做预览
- 读取模拟盘余额
- 自动把 `position / today_trades` 打印为 PDF 并归档
- 对账时优先消费最新 PDF
- 结构化保存当日信号、快照、对账、日志

现在额外补了一条研究主线：

1. `策略研究总览.py` 汇总当前基线、数据、优化、稳健性状态
2. `股票策略参数优化.py` 产出参数排行榜
3. `策略第二阶段稳健性诊断.py` 判断头部参数是否集中在同一路径
4. 确认通过后，再决定是否把新参数推进到交易链路

另外也已经为独立研究支线预留了一条入口：

1. `docs/因子研究支线任务书.md` 负责说明目标、边界、目录和验收标准
2. `factor_research/` 负责承接后续因子研究实现
3. 这条支线当前不接交易执行，不修改同花顺主流程

当前系统还不能稳定做到：

- 无人值守自动提交
- 自动化处理 `today_entrusts`
- 以程序自动对账完全代替人工确认

## 下一阶段建议

### 1. 先做轻量整理

- 统一入口
- 补齐 README
- 明确主流程和历史参考的边界
- 继续把文档和命令行习惯收口

### 2. 再做一次目录重组设计

- 判断哪些诊断脚本值得保留在根目录
- 哪些脚本应迁到 `tools/` 或 `diagnostics/`
- 重组前先确认引用路径、README、命令习惯是否都稳定

### 3. 最后再攻克剩余交易端问题

- `today_entrusts` 是否值得继续自动化
- 是否要为模拟盘增加“复核后一键状态同步”
- 是否要把自动导出接入稳定性巡检默认流程

当前这一项已经有可选实现：

- `交易_稳定性巡检.py --auto-export-pdf --pdf-printer "Microsoft Print to PDF"`

在这一步解决前，系统仍应保持“半自动”定位。
