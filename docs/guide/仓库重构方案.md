# 仓库重构方案

## 1. 背景

当前仓库已经从“单一回测脚本”演变成一个包含以下内容的混合型项目：

- 主线策略回测
- 参数优化与稳健性研究
- 同花顺半自动交易闭环
- 各类诊断脚本
- 因子研究支线
- 多轮过程性文档与对账文档

这些内容都在持续增长，但目前主要问题是：

1. 根目录脚本过多
2. 主入口、研究脚本、诊断脚本、历史参考混在一起
3. 文档缺少层级
4. 历史参考与当前主线边界不清
5. 新人或回看时很难快速判断“哪个是当前可信入口”

因此，本次重构的目标不是“改策略逻辑”，而是“重新整理项目结构和文件归属”。

## 2. 本次重构目标

本次重构只解决结构问题，不解决研究结论问题。

目标如下：

1. 让主线入口一眼可见
2. 把研究、诊断、历史参考、交易主链拆开
3. 把文档按角色分层
4. 为未来的项目字典、灰度接入、支线扩展预留稳定位置
5. 让 Git 提交更容易理解

## 3. 本次重构不做什么

这次重构不做以下事情：

1. 不修改交易执行语义
2. 不修改基线策略参数
3. 不调整因子研究结论
4. 不把实验策略直接接入交易执行
5. 不在同一轮中做大规模代码重写

换句话说：

- 先整理文件
- 再做后续工程改造

## 4. 重构原则

### 4.1 先软后硬

先完成：

- 目录设计
- 归类方案
- 文档导航

再进行：

- 物理迁移
- 兼容包装
- 清理归档

### 4.2 先保留兼容入口

对于当前已经常用的命令入口，不建议在第一轮直接删除。

更稳的做法是：

- 先迁移真实实现
- 根目录保留兼容 wrapper
- 等一段时间后再考虑删除旧入口

### 4.3 每次只迁一类

迁移顺序必须按类别拆开，避免一次性混乱：

1. 文档
2. 诊断脚本
3. 研究脚本
4. 数据脚本
5. 历史参考
6. 主入口

### 4.4 Git 提交必须可解释

每一轮迁移必须做到：

- 提交信息说明“迁了哪一类”
- 不把逻辑改动和结构改动混在同一 commit
- 不在工作区很脏的时候直接大搬家

## 5. 目标目录结构

建议目标结构如下：

```text
pythonProject/
├── README.md
├── app/
├── stock_trading/
├── research/
├── factor_research/
├── diagnostics/
├── scripts/
├── docs/
├── archive/
├── data/
├── outputs/
├── backups/
└── temp/
```

各目录职责如下。

### 5.1 `app/`

放主线可执行入口。

建议收纳：

- 交易总控入口
- 半自动交易入口
- 一键入口
- 稳定性巡检入口
- 可信基线回测入口

这类文件的特点是：

- 面向日常使用
- 面向主线流程
- 不能和研究脚本混放

### 5.2 `stock_trading/`

保留现有交易底层实现。

不建议本轮重命名。

因为这里已经具备：

- 状态
- 执行
- 信号服务

属于目前最稳定的一层。

### 5.3 `research/`

放主线研究脚本。

包括：

- 参数优化
- 稳健性诊断
- 回测诊断
- 主线研究总览

这部分仍属于主线，但不应和日常交易入口混在一起。

### 5.4 `factor_research/`

保留作为因子支线的代码域。

这里放：

- 因子研究基础模块
- 因子评估逻辑
- 因子标签
- 因子股票池

配套的因子研究脚本可考虑再细分为：

- `research/factor/`
或
- 保持根目录脚本一段时间后再迁入 `factor_research/entrypoints/`

第一轮不强行做深度拆分。

### 5.5 `diagnostics/`

放诊断脚本。

建议结构：

```text
diagnostics/
├── ths/
├── strategy/
└── data/
```

当前已经有：

- `diagnostics/ths/`

后续可继续扩展。

### 5.6 `scripts/`

放一次性辅助脚本、数据处理脚本、维护脚本。

例如：

- 下载数据
- 检查数据
- 构建股票分集

### 5.7 `docs/`

继续保留，但要分层。

建议结构：

```text
docs/
├── guide/
├── process/
├── research/
├── issues/
└── glossary/
```

### 5.8 `archive/`

放明确不再作为当前主线入口的文件。

例如：

- 旧版参考回测
- 旧版反演
- 被替代的历史对照脚本

这里的文件不是删除，而是明确标记为历史参考。

## 8. Git 整理策略

在真正开始物理迁移前，必须先满足两个条件：

1. 当前主线工作区相对干净
2. 因子支线新增文件已经先有一轮独立提交

否则会出现：

- 结构调整
- 研究新增
- 文档修订

三类变更混在一起，后续很难审。

建议 Git 策略如下：

### 第一步

先把当前尚未整理的新增研究文件、文档文件分别提交。

### 第二步

单开一个“结构整理”分支，例如：

```bash
git checkout -b chore/repo-structure
```

### 第三步

每一类迁移单独一个 commit。

例如：

1. `docs: group repository documents`
2. `chore: move ths diagnostics under diagnostics/ths`
3. `chore: move research scripts under research/`
4. `chore: archive legacy strategy files`
5. `chore: add compatibility wrappers for moved entrypoints`

## 9. 分阶段实施方案

### 阶段 A：文档先整理

只整理：

- `docs/`
- README 导航
- 项目导览

不动核心代码。

这是风险最低的一步。

### 阶段 B：诊断脚本归类

把诊断类脚本统一归入：

- `diagnostics/ths/`
- 需要时加 wrapper

这一步对交易主线影响较小。

### 阶段 C：研究脚本归类

把：

- 主线研究脚本
- 因子研究脚本

做第一次物理归类。

这一步开始要同步更新文档和入口。

### 阶段 D：历史参考归档

把不再作为主线使用的旧版脚本迁入 `archive/`。

这一步要特别小心，避免误归档仍被调用的文件。

### 阶段 E：主入口整理

最后才考虑是否把主入口迁入 `app/`。

如果要迁，建议：

- 保留根目录 wrapper
- 不立即删除旧入口

## 10. 当前是否是整理的时候

结论是：

- 是整理的时候
- 但不是“大搬家”的时候

当前最合理的节奏是：

1. 先定方案
2. 先把当前研究与主线改动分别提交
3. 再按阶段迁移

也就是说：

- 可以开始
- 但必须一步一步来

## 12. 一句话结论

仓库重构是合理且必要的。  
当前最好的做法不是立刻大规模移动文件，而是先按方案逐阶段整理。  
建议从“文档层整理”开始，再逐步推进到诊断脚本、研究脚本和历史归档。
