# 同花顺模拟盘问题记录

## 目标问题

当前需要解决的问题已经从“能否读到数据”切换为：

- 为什么持仓/成交有时可读、有时失败
- 查询读取不稳定到底受验证码、窗口状态还是控件结构影响
- 如何把当前能力从“半稳定读取”提升到“日常可依赖”

## 已验证事实

### 已成功

- 可以通过 `easytrader` 连接到同花顺模拟盘
- 可以读取 `balance`
- 可以读取 `raw_balance`
- 余额和股票市值变化可以反映模拟盘账户状态变化
- 32 位 Python 环境已经搭建完成
- 已经成功读取到 `raw_positions`
- 已经成功读取到 `raw_today_trades`
- 已经导出控件树、候选表格区域和 OCR 结果

### 已排除

- 不是 WSL 路径问题
- 不是单纯的 64 位 Python / 32 位客户端不匹配导致完全不可用
- 不是单纯字段映射过窄导致的全部误判

### 仍未解决

- 读取结果存在波动，不是每次都能稳定拿到持仓/成交
- `today_entrusts` 仍缺少足够样本验证
- 验证码识别成功率不高，仍可能拖慢甚至干扰查询流程
- 当只剩 OCR 兜底时，无法可靠恢复真实成交价和真实成本

## 当前推断

最可能的原因是以下之一：

1. 同花顺模拟盘查询页的表格控件结构与 `easytrader` 默认预期不完全一致
2. 已切到查询页，但不同候选 grid 的焦点和复制行为不稳定
3. 验证码识别失败会干扰查询动作链路
4. 窗口状态、分辨率、前台焦点会影响控件可操作性

## 当前策略

在完全稳定前，系统保持“半自动”：

- 程序负责：
  - 生成信号
  - 生成预览
  - 读取余额
  - 记录原始快照
  - 在必要时把 `cash / code / shares` 回填到本地状态
- 人工负责：
  - 模拟盘下单
  - 持仓和成交确认

## 下一步技术攻关建议

### 第一阶段：把问题看清

- 固定客户端版本、窗口状态、登录方式
- 每次对账时完整保存原始快照
- 记录是否弹验证码、是否多次失败后成功

### 第二阶段：看控件而不是只看返回值

- 用 pywinauto 检查查询页控件树
- 确认持仓/委托/成交页的表格控件类型、层级、控件 ID
- 判断是否需要放弃 `easytrader` 默认读取逻辑
- 当前控件诊断入口：
  [同花顺控件诊断.py](/mnt/d/量化/同花顺/pythonProject/同花顺控件诊断.py)

### 第三阶段：定制适配

- 如果默认 grid 不适配，单独实现：
  - 持仓页读取
  - 当日委托页读取
  - 当日成交页读取
- 当前已经加入 OCR 持仓兜底，但主路径仍优先使用原始表格读取
- 当前策略是不把 OCR 识别出的“成本价/成交价”当成真实成交依据

## 当前不建议做的事

- 在读取问题没解决前，直接推进无人值守自动下单
- 把偶发成功误判为“已经完全稳定”
- 把“读取为空”误认为“账户无持仓”
- 把 OCR 推断出的成本价误认为券商真实成交价
