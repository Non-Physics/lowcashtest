# 同花顺模拟盘问题记录

## 目标问题

当前需要解决的问题不是“能否连接同花顺”，而是：

- 为什么余额可读，但持仓/委托/成交长期为空
- 是否是客户端位数问题、控件结构问题，还是 `easytrader` 默认表格读取逻辑不适配

## 已验证事实

### 已成功

- 可以通过 `easytrader` 连接到同花顺模拟盘
- 可以读取 `balance`
- 可以读取 `raw_balance`
- 余额和股票市值变化可以反映模拟盘账户状态变化
- 32 位 Python 环境已经搭建完成

### 已排除

- 不是 WSL 路径问题
- 不是单纯的 64 位 Python / 32 位客户端不匹配导致完全不可用
- 不是字段映射过窄导致的误判
  - 已经落盘 `raw_positions / raw_today_entrusts / raw_today_trades`
  - 这些原始结果本身就是空

### 仍未解决

- `raw_positions = []`
- `raw_today_entrusts = []`
- `raw_today_trades = []`

即便在：

- Windows 原生环境
- 32 位 Python
- 已安装 `pywin32`
- 已安装 `pytesseract / tesseract`

的条件下，问题仍然存在。

## 当前推断

最可能的原因是以下之一：

1. 同花顺模拟盘查询页的表格控件结构与 `easytrader` 默认预期不同
2. 已切到查询页，但 grid 抓取逻辑没有抓到真正的表格数据
3. 模拟盘界面与真实券商下单界面在控件树上存在差异
4. 验证码与查询流程组合导致部分页面可见但表格未真正完成装载

## 当前策略

在问题解决前，系统保持“半自动”：

- 程序负责：
  - 生成信号
  - 生成预览
  - 读取余额
  - 记录原始快照
- 人工负责：
  - 模拟盘下单
  - 持仓和成交确认

## 下一步技术攻关建议

### 第一阶段：把问题看清

- 固定客户端版本、窗口状态、登录方式
- 每次对账时完整保存原始快照
- 记录是否弹验证码、是否多次失败后成功

### 第二阶段：看控件而不是只看返回值

- 用 pywinauto 检查查询页控件树
- 确认持仓/委托/成交页的表格控件类型、层级、控件 ID
- 判断是否需要放弃 `easytrader` 默认读取逻辑

### 第三阶段：定制适配

- 如果默认 grid 不适配，单独实现：
  - 持仓页读取
  - 当日委托页读取
  - 当日成交页读取

## 当前不建议做的事

- 在读取问题没解决前，直接推进无人值守自动下单
- 继续反复尝试仅靠 OCR 提高读取稳定性
- 把“读取为空”误认为“账户无持仓”
